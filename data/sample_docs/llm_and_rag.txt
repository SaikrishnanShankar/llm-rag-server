Large Language Models and Retrieval-Augmented Generation

Large Language Models

Large language models (LLMs) are neural networks trained on massive text corpora to understand and generate human language. Models like GPT-4, Claude, Llama, and Mistral contain billions of parameters trained using self-supervised learning on internet-scale datasets. They demonstrate emergent capabilities including reasoning, code generation, translation, and question answering.

Transformer Architecture
LLMs are built on the transformer architecture introduced in the 2017 paper "Attention is All You Need." Key components include multi-head self-attention mechanisms, feed-forward neural networks, positional encodings, and layer normalization. The attention mechanism allows the model to weigh the relevance of different parts of the input when generating each output token.

Pre-training and Fine-tuning
LLMs are pre-trained on large text corpora to learn general language representations. Fine-tuning adapts pre-trained models to specific tasks or domains using smaller labeled datasets. Instruction tuning trains models to follow natural language instructions. Reinforcement learning from human feedback (RLHF) aligns model outputs with human preferences.

Context Windows and Tokens
LLMs process text as tokens, which are subword units generated by tokenizers like Byte-Pair Encoding (BPE). Context windows define the maximum number of tokens the model can process at once. Modern LLMs have context windows ranging from 4K to 1M tokens. Longer context windows allow processing entire documents but require more compute.

Limitations of LLMs
LLMs have several inherent limitations. Knowledge cutoff: models only know information from their training data. Hallucination: models may generate plausible-sounding but incorrect information. Context limitations: information beyond the context window cannot be directly accessed. Static knowledge: without retrieval, models cannot access real-time information.

Retrieval-Augmented Generation (RAG)

RAG combines LLMs with information retrieval to overcome knowledge limitations. Instead of relying solely on parametric knowledge stored in model weights, RAG retrieves relevant documents and provides them as context to the LLM at inference time. This enables models to access up-to-date information, cite sources, and reduce hallucination.

RAG Architecture
A typical RAG system has two main phases. The indexing phase processes and stores documents: documents are split into chunks, chunks are converted to embeddings using an encoder model, and embeddings are stored in a vector database. The retrieval phase responds to queries: the query is converted to an embedding, similar chunks are retrieved from the vector database using approximate nearest neighbor search, and retrieved chunks are provided as context to the LLM along with the original query.

Vector Databases
Vector databases store high-dimensional embedding vectors and support efficient similarity search. Popular options include Pinecone, Weaviate, Qdrant, Milvus, and pgvector (PostgreSQL extension). They use approximate nearest neighbor algorithms like HNSW (Hierarchical Navigable Small World) and IVFFlat for fast retrieval. pgvector supports both exact and approximate nearest neighbor search directly within PostgreSQL.

Embedding Models
Embedding models convert text to dense vector representations. Sentence-transformers like all-MiniLM-L6-v2 produce 384-dimensional embeddings optimized for semantic similarity. Larger models like text-embedding-ada-002 produce 1536-dimensional embeddings with higher quality but more compute. The choice of embedding model significantly impacts retrieval quality.

Chunking Strategies
Document chunking strategy affects retrieval quality. Fixed-size chunking splits documents at regular character or token intervals, simple but may break semantic units. Sentence-based chunking preserves sentence boundaries, maintaining local coherence. Semantic chunking uses embedding similarity to find natural topic boundaries, producing more coherent chunks at higher computational cost. Chunk size and overlap are important hyperparameters affecting the specificity vs. coverage tradeoff.

Reranking
After initial retrieval, reranking improves result quality by applying a more sophisticated model to score and reorder candidate chunks. Cross-encoder models consider both query and document together, unlike bi-encoder models used for initial retrieval. Reranking adds latency but often significantly improves answer quality. ColBERT and other late-interaction models offer a middle ground between bi-encoder speed and cross-encoder quality.

Advanced RAG Techniques
HyDE (Hypothetical Document Embeddings) generates a hypothetical answer to the query and uses its embedding for retrieval, often improving relevance. Query decomposition breaks complex questions into sub-questions, retrieves for each, then combines results. Multi-hop retrieval chains retrieval steps for complex reasoning tasks. Self-RAG teaches models to adaptively retrieve and critique their own outputs.

Evaluation of RAG Systems

RAGAS (Retrieval-Augmented Generation Assessment) is a framework for evaluating RAG pipelines. Key metrics include faithfulness (whether the answer is supported by retrieved context), context relevance (whether retrieved chunks are relevant to the query), and answer relevance (whether the answer actually addresses the query). These metrics can be computed automatically using LLMs, enabling large-scale evaluation without human annotation.

vLLM and LLM Serving

vLLM is a high-throughput LLM inference and serving library. Key innovations include PagedAttention, which manages attention key-value cache memory efficiently, enabling much higher throughput than naive implementations. Continuous batching dynamically groups requests to maximize GPU utilization. vLLM provides an OpenAI-compatible API, making it easy to swap between providers. It supports models including Llama, Mistral, Falcon, and many others.

LangChain and LangGraph

LangChain is a framework for building applications with LLMs. It provides abstractions for chains, agents, and retrieval. LangGraph extends LangChain with graph-based agent workflows, enabling stateful multi-actor applications. Nodes in a LangGraph represent computation steps, edges define the flow between steps, and state is passed between nodes. LangGraph supports cycles, conditional routing, and parallel execution, making it suitable for complex agentic workflows.

MLflow for ML Experiment Tracking

MLflow is an open-source platform for managing the ML lifecycle. It tracks experiments by logging parameters, metrics, and artifacts. The MLflow Model Registry stores and versions models. MLflow Projects define reproducible ML code. MLflow Serving deploys models to various serving frameworks. For RAG systems, MLflow can track experiments comparing different chunking strategies, embedding models, and retrieval configurations.
